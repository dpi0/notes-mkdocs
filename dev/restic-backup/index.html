    Note

    REST server is better than SFTP transfer

    1. faster and more scalable
    2. adds the optional ability to run in append-only mode, prevents an attacker from wiping your server backups when access is gained to the server being backed up.
    3. really simple implementation and as such could be used on the low-end devices
    4. in some cases, for example behind corporate firewalls, HTTP/S might be the only protocol allowed

    in my testing, REST is about 20-25% faster than SFTP on a 14G (~15,000 files) transfer (54 sftp vs 42 min on https)

    although it takes a second to setup, REST is clearly better (maybe without the HTTPS overhead it could be faster)

    using REST server¶

    • setup the server using the compose file (NOTE the syntax for the environment var, key: value. the key=value syntax doesn't work here fsr)
    • also the flags used in compose are --tls --append-only where tls enables TLS and append-only allows creation of new backups but prevents deletion and modification of existing backups. This can be useful when backing up systems that have a potential of being hacked.
    • to backup using plain HTTP (not recommended, also remove the tls flag for this)
    # create admin user
docker exec -it restic_server create_user admin
# enter password and save this

# this is for the auth to the rest server (the user password couldn't find a way to have a password file)
export RESTIC_REST_USERNAME=admin; export RESTIC_REST_PASSWORD="z59FxsYKn70eS0mp2blv9iY9UGsuasK7"
# this is for the repo
export RESTIC_REPOSITORY="rest:http://server:5050/repo_name"; export RESTIC_PASSWORD_FILE=$HOME/.password_restic
restic init -r $RESTIC_REPOSITORY
# this creates the repo under mounted `/data` dir, so choose this location carefully
# also the server verifies the `admin` user using the `.htpasswd` placed under `/data`
    • to backup using HTTPS:
    # ON SERVER
# generate pub/priv keys
LOCATION="$HOME/restic-rest-server-backup"

sudo openssl req \
-newkey rsa:4096 \
-sha512 \
-x509 \
-days 365 \
-addext "subjectAltName = IP:127.0.0.1,DNS:dpi0.cloud" \
-nodes \
-keyout $LOCATION/private_key \
-out $LOCATION/public_key

# the fields are all optional, but is nice to fill email, name, country, state
# -nodes (removes password from keys, restic doesn't work with keys with passwords), days 365 (validity of the cert), -addext "subjectAltName" (is optional)

unset LOCATION

# restart server
docker compose down; docker compose up -d

# create admin user
export REST_SERVER_USERNAME="admin"
export REST_SERVER_PASSWORD="$(< /dev/urandom tr -dc 'A-Za-z0-9' | head -c32)"
docker exec -it restic_server create_user $REST_SERVER_USERNAME $REST_SERVER_PASSWORD
echo $REST_SERVER_PASSWORD
unset REST_SERVER_PASSWORD
docker compose down; docker compose up -d
docker logs restic_server

# ON CLIENT
export RESTIC_REST_USERNAME=admin
export RESTIC_REST_PASSWORD="<the_password_set_on_server_for_the_user>"
export RESTIC_REPOSITORY="rest:https://server:5050/REPO_NAME"; export RESTIC_PASSWORD_FILE=$HOME/.password_restic

restic init -r $RESTIC_REPOSITORY --insecure-tls
# --insecure-tls because these are self signed certs

restic snapshots --insecure-tls
restic stats --insecure-tls

unset RESTIC_REST_PASSWORD

# using my custom function to clear the password variable from `~/.zsh_history` file as well
remove_variable_from_history "RESTIC_REST_PASSWORD"

    Warning

    getting a signed certificate from Let's Encrypt requires your domain to be pointing to a PUBLIC IPv4 Address e.g a public A record

    so dpi0.cloud currently won't work as it points to local ip

    have to use self signed certs

    Tip

    useful when using the rest-server binary and don't have htpasswd binary

    create a .htpasswd file without the httpasswd tool with a random password using md5 hash

    `username="admin"; password=$(</dev/urandom tr -dc 'A-Za-z0-9' | head -c32); echo "$username:$(echo -n "$password" | openssl passwd -apr1 -stdin)" > backup/.htpasswd`
echo $password
unset password

    but for restic rest server we need to use bcrypt so have to use python + bcrypt + docker (openssl doesn't support brcrypt now)

    # Dockerfile
FROM python:3
RUN pip install bcrypt

    docker build -t bcrypt-generator .
export ADMIN_PASSWORD="7jzgikv0DbuGZWr2JquL8U2UTdETS79v"
export ADMIN_USERNAME="admin"
docker run --rm bcrypt-generator python -c "import bcrypt; print(bcrypt.hashpw(b'$ADMIN_PASSWORD', bcrypt.gensalt(rounds=5)).decode('utf-8'))" > hash.txt
echo "$ADMIN_USERNAME:$(cat hash.txt)" > .htpasswd
unset ADMIN_PASSWORD
remove_variable_from_history ADMIN_PASSWORD

    Backrest¶

    • whatever device contains the source files to be backed up, needs to run the backrest container.
    • you can then specify the repo location in the backrest gui, either local or rest or sftp or s3 etc

    for rclone in backrest container

    echo "**/rclone.conf" >> $HOME/home-server/.gitignore

cat $HOME/.config/rclone/rclone.conf -p
# either cp the entire file, or only the necessary rclone account config

# confirm
docker compose down backrest; docker compose up -d backrest
docker exec -it backrest cat /root/.config/rclone/rclone.conf

    Forget and Prune Policy¶

    say i backup every 2 days. my forget/prune policy

    • Keep hourly for 8 hours
    • Keep daily for 7 days
    • Keep weekly for 4 weeks
    • Keep monthly for 3 months

    is set to run every 2 weeks.

    what will be left when the policy runs every 2 weeks?

    i will have roughly 7 snapshots by the end of a 2-week period

    • hourly 8: won't apply, no hourly snapshots were taken.
    • daily 7: won't apply, as keep 1 snapshot per day for last 7 days. multiple snapshots were not ttaken daily.
    • weekly 4: in two weeks (forget policy time period) as i'm taking snapshot every 2 days. every week 1 will have about 2-3 snapshots. for the last 4 weeks, restic will 1 snapshot per week.
    • monthly 3: keep 1 snapshot per month, for the last 3 months.

    what will be kept:

    Time Period Backup Frequency
    Last 7 days Every backup (every 2 days)
    2 to 4 weeks One snapshot per week
    Last 3 months One snapshot per month
    Beyond 3 months No snapshots retained

    GPT recommends this policy and time period if i'm taking backup every 3 days.

    • Keep daily for 3 days
    • Keep weekly for 4-6 weeks
    • Keep monthly for 3-6 months
    • (Optional): Keep yearly for 1-2 years

    And you would run this policy once every month.

    Reason:

    • Keeping snapshots for 3-4 days ensures you’ll retain the latest snapshot if you need quick recovery within a week.
    • Keeping one snapshot per week over the last month to month and a half provides enough snapshots to recover from any issues that arise during recent weeks.
    • Retaining one snapshot per month for up to 6 months lets you keep long-term versions, while pruning older snapshots that take up space
    • If you want some annual retention for critical records or legal compliance, consider keeping one snapshot per year for up to 2 years

    GPT suggested this for a backup every day at 11AM

    • Keep hourly for 6 hours
    • Keep daily for 7 days
    • Keep weekly for 4 weeks
    • Keep monthly for 6 months
    • (Optional): Keep yearly for 2 years

    And run this policy every week.
