    1. Managing External Drives

    1.0.1 format, label and name external drivesÂ¶

    refer: https://wiki.archlinux.org/title/File_systems

    Note

    Choosing b/w file system for a USB Drive, it's NTFS vs exFAT (ext4, BTRFS ZFS NOT MEANT FOR EXTERNAL)

    If truly "cross-platform", exFAT (as mac can't write to NTFS). If it it's just Windows and Linux, NTFS is fine.

    choose exFAT only for simple file sharing not for backups. and only for SSD not HDD.

    Extended File Allocation Table, or exFAT, was also introduced by Microsoft. replacement for FAT32

    Linux has support for both exFAT and NTFS. The problem is that neither of those filesystems were designed for a UNIX-like system such as Linux.

    but exFAT is prone to corruption as no journaling. doesn't support UNIX file permissions. Designed for flash drives with unimportant temporary data and compatibility with embedded systems

    Note

    for an external HDD

    if plugging into a linux server OR primarily will be used with linux ext4 (don't understand enough about BTRFS or XFS or ZFS for now so avoid)

    if need cross-compatibiliy then NTFS (read only mac). avoid exFAT for external drives especially HDD only designed for small solid storage devices.

    for an external SSD

    same reasons for ext4 and NTFS

    TOOLS:

    • fdisk designed primarily for MBR (limited GPT support) avoid
    • parted general purpose, best for scripting, supports both MBR and GPT, lacks some GPT-specific features offered by gdisk.
    • gdisk only for GPT. use this when not scripting

    Note

    the file system overhead for ext4 is 5%. so for 931.5 GiB, about 870G is actually available!

    1.0.1.2 1. completely clear the driveÂ¶

    https://wiki.archlinux.org/title/Securely_wipe_disk

    sudo umount /dev/disk/by-label/LABEL

# quick format (NOT REALLY A WIPE, will only clear the file system and files on the chosen partition, DOES NOT CLEAR PARITION TABLE)
sudo mkfs.ext4 /dev/disk/by-label/LABEL (or mkfs.exfat, ntfs etc)

# overwrite with zeros (secure, but slow as each bit goes to 0)
sudo dd if=/dev/zero of=/dev/disk/by-id/NAME bs=1M status=progress
# what it does is:
# copies over endless stream of zeros 1MB at a time till the disk fills up, essentially wiping the disk
# changing bs value to say 10M will copy over 10M of zeros at a time which is faster but not very precise, smaller bs value like 1K is very very slow but very precise
# you can add a count value to specify the no. of blocks to copy over, like count=1000 will copy over 1G of data only and not fill the entire disk
# by default no count value is specified so the command fills up the entire disk (if 931G drive, then will take a long time with 40MB/s will take 5.5 hours)
# say count=50000 meaning 500G of zero data blocks will be transferred, so the first 500G of the drive will be wiped with zeros. NOT THE ENTIRE DRIVE.

# overwrite with random data (more secure but really slow, each bit gets random bit)
sudo dd if=/dev/urandom of=/dev/disk/by-id/NAME bs=1M status=progress

# safest way (but the slowest too as it does multiple passes)
shred -v /dev/disk/by-id/NAME
# for only 1 pass
shred --verbose --random-source=/dev/urandom -n1 --zero /dev/disk/by-id/NAME

# verify the wipe
sudo fdisk -l /dev/disk/by-id/NAME

    Do not overwrite an SSD with random data if you plan to use TRIM

    1.0.1.3 2. add the partition table (GPT or MBR)Â¶

    sudo umount /dev/disk/by-id/NAME_OF_THE_DRIVE

sudo gdisk /dev/disk/by-id/NAME
# o (create a new empty GPT partition table)
# w (write the changes and exit)

    MBR still used today. found on older hardware and systems that do not support UEFI. often sufficient for smaller drives (under 2 TB)

    USB drives are formatted with MBR, especially if they are intended for use with older systems, bootable rescue disks, or installation media for older operating systems.

    supports a maximum disk size of 2 TB and can only handle up to four primary partitions.

    does not have redundancy built into its structure, which means that if the MBR is corrupted, the entire partitioning scheme may be compromised.

    lacks advanced features offered by GPT, such as partition alignment, larger partition sizes, and partition type GUIDs.

    use GPT for boot drives or when redundancy and data integrity are essential, large disks over 2 TB, requiring many partitions (more than four), better performance on SSDs due to partition alignmentm.

    1.0.1.4 3. add partitionsÂ¶

    # here i will have 1 partition for the entire drive
sudo gdisk /dev/disk/by-id/NAME
# n (create a new partition)
# Accept the default partition number (usually 1)
# Accept the default values for the first and last sectors to use the entire disk.
# simply for a linux specific drive **8300** (ext4, ZFS, BTRFS) for Microsoft's data tables NTFS or exFAT **0700**
# w (write the changes and exit)

# confirm the changes
sudo gdisk /dev/disk/by-id/NAME
# p (to print the partition table)

    partitions are logical division within a storage device. treated as a separate "drive" by operating systems.

    Each partition can then be formatted with a file system and mounted separately.

    1.0.2 5. verifyÂ¶

    Caution

    please use /dev/disk/by-id/NAME_OF_THE_DRIVE and not LABEL when verifying integrity of partitions or checking partition table.

    as LABEL refers to PARTITIONS, NOT entire DRIVES on these tests on parted or gdisk causing confusion. like saying parition has a partition table of loop (when in reality partition table is not inside the label/partition BUT on the whole drive/by-id)

    when simply mounting/umounting, only then use LABELS

    # check File System Integrity
sudo fsck.exfat /dev/disk/by-id/usb-TOSHIBA_MQ01ABD100_235678C218CA-0:0
sudo fsck.ext4 /dev/disk/by-id/NAME
sudo ntfsfix /dev/disk/by-id/NAME

# verify partition table, partitions and file systems
lsblk -f
sudo parted /dev/disk/by-id/NAME print
sudo fdisk -l /dev/disk/by-id/NAME

# mount
sudo mkdir -p /mnt/32G
sudo mount /dev/disk/by-label/32G /mnt/32G

    1.0.4 auto mount drives/partitions on bootÂ¶

    https://wiki.archlinux.org/title/Fstab

    UUID=abc /     ext4   defaults  0      1
UUID=xyz /home ext4   defaults  0      2

    is checked by the dump(8) utility. This field is usually set to 0, which disables the check.

    sets the order for file system checks at boot time; see fsck(8). For the root device it should be 1. For other partitions it should be 2, or 0 to disable checking.

    1.0.4.1 external devicesÂ¶

    https://wiki.archlinux.org/title/Fstab#External_devices

    # get the UUID
lsblk -f

sudo vim /etc/fstab
---
$UUID    /hdd    ext4    defaults,nofail,x-systemd.device-timeout=5    0 2
---

sudo systemctl daemon-reload
# onlt then reboot

    x-systemd.device-timeout=5 is a systemd-specific option that sets a timeout for how long systemd should wait for the device to become available before giving up.

    The nofail option is best combined with the x-systemd.device-timeout option. This is because the default device timeout is 90 seconds, so a disconnected external device with only nofail will make your boot take 90 seconds longer, unless you reconfigure the timeout as shown.

    Make sure not to set the timeout to 0, as this translates to infinite timeout.

    1.0.4.2 recommended mount options by Bad3rÂ¶

    https://github.com/Bad3r/Optimal-Storage-Mount-Options/blob/main/README.md#recommended-mount-flags-ext4

    https://reddit.dpi0.cloud/r/sysadmin/comments/1fjamf0/recommended_mount_flags_for_storage_devices_on/

    defaults,relatime,barrier=1,data=ordered,errors=remount-ro,commit=60

    options explained: https://github.com/Bad3r/Optimal-Storage-Mount-Options/blob/main/README.md#options-explanation

    • defaults (recommended and fine)

    • rw: Mounts the filesystem as read-write.

    • suid: Allows execution of binaries with setuid bits.
    • dev: Allows character and block devices to be created on the filesystem.
    • exec: Allows execution of binaries on the filesystem.
    • auto: Allows the filesystem to be mounted automatically with the -a option of mount.
    • nouser: Prevents regular users from mounting the filesystem.

    • async: Writes to the filesystem are asynchronous, meaning they are not immediately flushed to disk.

    • barrier=1 (helps protect against data corruption in case of sudden power loss.)

    Enables write barriers, which ensure that write operations are properly ordered with respect to power loss.

    This is particularly important for data integrity.

    • data=ordered (good option for data integrity)

    Ensures that data is written to disk in an ordered manner. Specifically, it guarantees that all data is flushed to disk before the metadata (e.g., file size, timestamps) is updated.

    • errors=remount-ro (helps protect against data corruption in case of filesystem errors.)

    If the filesystem encounters an error, it will remount the filesystem as read-only. This prevents further writes to the filesystem, which can help prevent data corruption.

    • commit=60 (value of 60 seconds (1 minutes) default=5 seconds is a reasonable balance between performance and data integrity)

    Sets the interval (in seconds) at which the filesystem commits metadata changes to disk.

    this comes with a risk: any power loss or disconnection means up to 1 minute of unsaved data may be lost.

    For HDDs that arenât frequently disconnected, this option can be an effective balance of performance and data safety.

    However, for highly critical data, a shorter commit interval would be safer.

    1.0.6 LUKS Encryption on an external driveÂ¶

    sudo dd if=/dev/zero of=/dev/disk/by-id/NAME bs=1M status=progress

sudo gdisk /dev/disk/by-id/NAME

sudo gdisk /dev/disk/by-id/NAME

sudo gdisk /dev/disk/by-id/NAME

sudo cryptsetup luksFormat /dev/disk/by-id/NAME

sudo cryptsetup open /dev/disk/by-id/NAME TEMP_MOUNT_NAME

sudo mkfs.ext4 /dev/mapper/TEMP_MOUNT_NAME

sudo mkdir -p /mnt/TEMP_MOUNT_NAME
sudo mount /dev/mapper/TEMP_MOUNT_NAME /mnt/TEMP_MOUNT_NAME

sudo umount /mnt/TEMP_MOUNT_NAME
sudo cryptsetup close TEMP_MOUNT_NAME

    on a new desktop

    export DRIVE=4G
sudo cryptsetup open /dev/disk/by-id/NAME $DRIVE

sudo mkdir -p /mnt/$DRIVE
sudo mount /dev/mapper/$DRIVE /mnt/$DRIVE

sudo umount /mnt/$DRIVE
sudo cryptsetup close $DRIVE
