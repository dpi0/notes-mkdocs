    efi vs esp vs bootloaders vs firmware¶

    https://www.rodsbooks.com/efi-bootloaders/principles.html

    Power on -> UEFI initializes hardware -> POST --> UEFI firmware loaded from ROM on mb --> UEFI scans ESP -> Finds bootx64.efi/grubx64.efi -> Loads GRUB via /boot/grub/grub.cfg -> GRUB loads kernel -> Kernel initializes system (more after this below)

    • EFI is the standard, and UEFI is an implementation.
    • firmware: UEFI, BIOS Low-level software embedded in hardware. Initializes hardware during the booting process
    • esp: the partition containing .efi files, necessary for the firmware to boot the operating system.
    • bootloaders: GRUB, systemd-boot, rEFInd etc. load the operating system kernel and initialize the OS. providing options to select different kernels or OSes.

    Linux Kernel¶

    core component of the Linux operating system

    main components are:

    • Process Management: scheduling, creation, and termination.
    • Memory Management: allocation and deallocation of memory, including virtual memory management.
    • File Systems
    • Device Drivers: providing a standardized way for the system to communicate with them.
    • Networking
    • Security: user permissions, SELinux, and other access controls.

    initramfs¶

    https://wiki.archlinux.org/title/Arch_boot_process#initramfs

    temporary root file system loaded into memory during the boot process.

    provides the necessary drivers and tools to mount the actual root file system

    bootloader loads the kernel and initramfs into memory. The kernel then uses initramfs to initialize the system and mount the real root file system.

    initramfs-linux-fallback.img: Larger fallback initramfs image with more drivers for recovery.

    kernel and the initial RAM filesystem (initramfs) are typically located in the /boot

    • An initramfs (initial ram file system) is used to prepare Linux systems during boot before the init process starts.
    • initramfs, which is kind of a mini-system that runs before your "real" system and is responsible for bringing that up
    • checks whether your root filesystem is corrupt or not before attempting to mount it and pivot into it.
    • If your filesystem is corrupt for whatever reason, it refuses to mount it and instead drops you into an environment
    • where you can do whatever actions to fix the corruption (in your case, running fsck was enough).

    the binaries are:

    Kernel: vmlinuz-linux, vmlinuz-linux-lts, etc. usually around 13-14M compressed 100-150MB uncompressed
    Initramfs: initramfs-linux.img, initramfs-linux-lts.img, etc.

    .rwxr-xr-x 13M root 30 Jun 05:45 ├── vmlinuz-linux
    .rwxr-xr-x 14M root 30 Jun 21:15 └── vmlinuz-linux-zen

    .rwxr-xr-x 153M root 30 Jun 01:54 ├── initramfs-linux-fallback.img
    .rwxr-xr-x 43M root 30 Jun 01:54 ├── initramfs-linux.img
    .rwxr-xr-x 8.1M root 1 Jun 11:46 ├── intel-ucode.img

    in /boot/grub/grub.cfg you would see menuentry and what to do when used that option

    • Once the kernel is loaded into memory, GRUB transfers control to it.
    • kernel begins execution and initializes the core system components.
    • kernel sets up basic hardware components (CPU, memory management, interrupt controllers, etc.).
    • decompresses the initramfs into a temporary filesystem in RAM.
    • initramfs contains drivers and scripts needed to initialize hardware that is critical for mounting the real root filesystem. useful for loading drivers for disk controllers, filesystems, and other hardware necessary to access the root filesystem.
    • Once the root filesystem is mounted by initramfs, the kernel pivots from the initramfs to the real root
    • The kernel then looks for the init process (usually /sbin/init or /usr/lib/systemd/systemd)
    • The init process is responsible for starting all other processes.

    GPT vs MBR¶

    GPT

    • modern partitioning scheme introduced as part of the UEFI standard.
    • supports virtually unlimited partitions (128 in Windows) and partitions larger than 2 TB.
    • UEFI firmware reads the GPT to find the bootloader on the ESP.
    • Designed to work with UEFI firmware but can be used with BIOS using hybrid MBR/GPT setups.
    • faster, more robust with redundancy and checksums for partition tables.

    MBR 💀

    • legacy partitioning scheme used in older systems.
    • up to 4 primary partitions, each up to 2 TB.
    • BIOS reads the MBR, which contains the bootloader. and works ONLY with BIOS firmware.

    Swap¶

    • Modern systems show minimal performance difference between swap files and swap partitions.
    • Swap files offer greater flexibility and are easier to manage than partitions.

    with hiberation: keep swap 1x to 1.5x of RAM

    Suspend to RAM (S3) vs. Suspend to Idle (S0)¶

    S0 is the modern nightmare/devil

    • Low power idle state, system remains semi-active.
    • Higher power consumption, almost instant wake-up.

    S3 is the better harder stronger not faster though :(

    • Low power state, saving system context to RAM.
    • Very low power consumption, longer wake-up time.

    Firmware¶

    It's firm, because it sits between soft and hard 🤯

    It's not usually an entire operating system, but a single program that can do things like see if the rest of the computer is okay and look through other devices to find the next program to run.

    There's typically no "process" as there is either no kernel (no need for anything else to run) or a very minimal one as it directly controls the hardware.

    As for how it's started, most CPUs will begin executing code from a fixed memory address as part of the design. On x86 this is 0xFFFFFFF0

    Most modern firmware uses variations of Flash memory technology. electrically erasable and reprogrammable non-volatile

    NOR Flash: Commonly used for firmware storage in embedded systems and microcontrollers. It supports random access, making it suitable for code execution.

    NAND Flash: Used in solid-state drives (SSDs) and storage devices due to higher storage density.

    other variations of firmware are from best to worst:

    • EEPROM (Electrically Erasable Programmable Read-Only Memory): non-volatile memory that can be electrically erased and reprogrammed multiple times. Can be selectively erased and reprogrammed using electrical signals, offering flexibility compared to EPROM.
    • EPROM (Erasable Programmable Read-Only Memory): non-volatile memory that can be erased and reprogrammed using ultraviolet (UV) light exposure. less convenient than EEROM or EEPROM.
    • EEROM (Electrically Erasable Read-Only Memory): non-volatile memory that allows for electrical erasure and reprogramming. It combines features of both ROM and EPROM.
    • ROM: Traditional ROM is non-volatile memory that is programmed during manufacturing and cannot be modified afterward. Not erasable or reprogrammable after manufacturing.

    what firmware contains:

    • UEFI/BIOS: Initializes and tests hardware components during the boot process. Loads Kernel. Configures hardware settings like CPU, memory, and peripherals.
    • Embedded Controller (EC) Firmware: Manages power functions, keyboard input, battery charging, and other low-level tasks.
    • Peripheral Firmware: Firmware for integrated devices like network interfaces, audio controllers, and storage controllers.

    UEFI vs BIOS¶

    coolest article on this: https://www.happyassassin.net/posts/2014/01/25/uefi-boot-how-does-that-actually-work-then/

    UEFI

    • A modern replacement for BIOS. Introduced to address limitations of BIOS and provide more advanced features.
    • Supports larger storage devices, faster boot times, and a more flexible pre-boot environment.
    • Uses the GPT (GUID Partition Table) partitioning scheme and can access drives larger than 2 TB. UEFI firmware includes its own boot manager.
    • Graphical and text-based interface with a more user-friendly setup screen
    • Faster boot times and includes features like Secure Boot
    • Can boot from both MBR and GPT drives

    BIOS 💀

    • For most practical purposes, especially on modern systems, you can focus on UEFI. Most new motherboards and PCs come with UEFI firmware and may not even support traditional BIOS.
    • The traditional firmware interface for PCs. Originated in the 1980s and has been used for decades.
    • Executes a series of instructions (POST) to initialize hardware and boot the operating system.
    • Many users and manufacturers still refer to UEFI as "BIOS" due to historical reasons and familiarity.
    • The setup utility of UEFI firmware is often called "BIOS setup" in documentation and on-screen prompts. for no fucking reason. This led to confusion as users had to understand both systems during the transition period.

    Modern Linux Physical Security Measures¶

    first checkout https://wiki.archlinux.org/title/Security

    • you could lock down UEFI with a password, but that can be easily mitigated by resetting the CMOS battery
    • disk encryption like LUKS helps when someone can get yo ssd and plug it into theirs, no user passwd will protect ya
    • disable root account
    • pam_faillock.so is enabled by default to lock out users for 10 minutes after 3 failed login attempts in a 15 minute period on arch
    • use a strong user password, and prevent brute force via passwd and PAM OR just use fail2ban

    Opening CMD or PowerShell as Admin¶

    1. Press Win + R to open the Run dialog.
    2. Type cmd for Command Prompt or powershell for PowerShell.
    3. Instead of pressing Enter, press Ctrl + Shift + Enter to open as Administrator.

    Checking Admin Privileges¶

    To verify if Command Prompt or PowerShell is running with Administrator privileges, use the following command:

    net session
