    1.2.2 DNS Configuration

    More info: AdGuardHome FAQ

    • By default, port :53 is occupied by systemd-resolved (sudo lsof -i :53).
    • Best solution:
    • Disable systemd-resolved. Note: If systemd-resolved is not running, /etc/resolv.conf should contain the DNS servers.
    • Contents of resolv.conf take over after stopping syst*md-resolved.
    • Ensure systemd-networkd is running; it controls internet access and uses netplan for configuration.
    • systemd-networkd handles IP addressing, routes, and DNS configuration.
    • systemd-resolved provides DNS resolution and caching, listening on 127.0.0.53 and forwarding to upstream DNS servers.
    • Changing AdGuard to port 66 requires setting 127.0.0.1:66 in resolv.conf and router configuration. It's easier to disable resolved and use port 53 with AdGuard.

    1.2.3 Configuring Redirect from *.home to *.xxxxx.duckdns.org

    • In filters > DNS rewrites, add rule *.home -> server.
    • Verify DNS records with dig files.home (should point to server in its A record).
    • Update Caddyfile for HTTPS redirection.
    • Note: At the start of each browser/server session, you may need to accept the security warning.
    • For a smoother experience, buy a hillu.home domain with a valid TLS certificate for the Caddyfile.
    • Alternatively, use a self-signed certificate installed on each device for *.home.
    • or better just use cloudflare tunnels

    1.4 rsync

    • scp is outdated/has issues since 2019.

    even locally use rsync instead of cp

    rsync -havz --progress /path/to/source/ /path/to/destination/
rsync -r -havz --progress /path/to/source/ /path/to/destination/

    1.4.1 Common rsync Options

    • -a: archive (preserve permissions, links, attributes)
    • -z: compress
    • -r: recursive (directory)
    • -v: verbose
    • -h: human-readable units
    • -e: use SSH

    1.4.2 Setup SSH Agent

    eval "$(ssh-agent -s)"
ssh-add ~/.ssh/ubuntu-server

    1.4.3 send files --> to remote

    rsync -havz --progress -e "ssh -p 3333 -i ~/.ssh/private-key" file1 file2 dpi0@server.home:/home/dpi0/folder/

    1.4.4 send dirs --> to remote

    rsync -r -havz --progress -e "ssh -p 3333 -i ~/.ssh/private-key" dir1 dir2 dpi0@server.home:/home/dpi0/apps/

    1.4.5 send all files in a dir --> to remote

    Change dir1 to dir1/ to send all files in the directory.

    1.7 NFS

    Video Guide

    NFS Near-native speed/performance, primarily Linux-to-Linux SAMBA Default encryption, multi-user support, cross-platform.

    • Not encrypted by default but can be.
    • No user authentication by default; access is restricted by IP address/hostname.
    • Use Kerberos for stronger authentication.
    • Requires matching user/user group IDs on both client and server.
    • Lower latency compared to SAMBA.
    • With autofs, no need for manual mount/umount, but mainly for desktops.

    1.7.1 on the server

    sudo apt update
sudo apt install nfs-kernel-server -y

sudo cp /etc/exports /etc/exports.bak

## format: directory_to_share    client(share_option1,...,share_optionN)

sudo tee -a /etc/exports <<EOF
/data/files/nfs/pictures                192.168.1.80/24(rw,no_subtree_check)
/data/files/nfs/documents               192.168.1.80/24(rw,no_subtree_check)
/data/media/films/dpi0              192.168.1.80/24(rw,no_subtree_check)
/data/media/tv/dpi0         192.168.1.80/24(rw,no_subtree_check)
EOF

sudo systemctl restart nfs-kernel-server
sudo systemctl status nfs-kernel-server

    1.7.2 on the client

    1.7.2.1 ubuntu

    sudo apt update
sudo apt install nfs-common

    1.7.2.2 arch

    sudo pacman -S nfs-utils

sudo mount server:/data/files/nfs/documents /data/nfs/documents

## check mount
df -h

sudo umount /data/nfs/documents

    to automount on demand, and umount automatically

    either use autofs or better systemd umount or even a startup command in hyprland.conf

    echo "server:/data/files/nfs/documents  /data/nfs/documents  nfs  noauto,x-systemd.automount,x-systemd.idle-timeout=10min  0  0" | sudo tee -a /etc/fstab
sudo reboot
    • noauto: Prevents the file system from being mounted automatically at boot time.
    • x-systemd.automount: Tells systemd to automatically mount this file system on access.
    • x-systemd.idle-timeout=10min: Unmounts the file system after being idle for 10 minutes. Adjust the timeout as needed.

    1.11 unbound

    https://docs.pi-hole.net/guides/dns/unbound/#pi-hole-as-all-around-dns-solution

    https://t.ly/TO4Re **best technical discussion on ubound vs quad9**

    https://t.ly/RNFu6

    https://t.ly/VatQn

    https://t.ly/1d1RA

    https://ccut.link/CjF4othc best technical answer

    • Unbound isn't encrypted by default but supports DoT/DoH.
    • Unbound recursive ensures all queries stay local, eliminating third parties, unlike DoT/DoH which requires a third-party provider.
    • Unbound can function as a forwarding resolver with TLS but not as a recursive resolver with TLS. Details
    • DNSSEC in Unbound verifies query integrity.
    • Unbound's privacy comes from avoiding third parties, not encryption.
    • DNS encryption adds little privacy; your final destination is visible.
    • With Unbound, ISPs can see and log your DNS requests, which are sent in plain text.
    • DoT/DoH encrypts data in transit but requires trusting third-party providers like NextDNS, Quad9, or Cloudflared.
    • Unbound forwards DNS requests in clear text to nameservers, mixing with other queries, reducing anonymity.
    • Configuring Unbound as a forwarding resolver defeats its purpose as a recursive resolver.
    • At least one third-party can see your DNS requests; recursive vs encrypted reduces one risk category.
    • Advanced DNS setups make IP and behavior fingerprinting harder but don't provide anonymity. For true anonymity, use TOR, despite its flaws.

    1.12 Port Forwarding

    Port forwarding allows a device on your network (e.g., a server) to be accessible from the internet.

    • Requirement: Your router must have a public IP address.
    • India: ISPs often use CGNAT, placing your router inside the ISP's NAT, preventing direct access.
    • US: Most ISPs do not use CGNAT.
    • Without a public IP, accessing internal services like Caddy/NGINX from outside the network isn't possible.

    1.12.1 Solutions

    1. Static IP: Obtain from your ISP, usually expensive.
    2. WireGuard: Host on a cloud VPS and route traffic back to your network.
    3. Cloudflare Tunnels: Simplest solution. More info: Awesome Tunneling Recommendations
