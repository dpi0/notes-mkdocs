    Ubuntu Home Server Install

    configure internet¶

    during installation¶

    • sometimes wifi wlp2s0 or wlp3s0 whatever doesn't show up, keep going back and forth in the install AND it WILL show up!
    • then edit wifi/eth and connect to the internet
    • configure static ip from DHCP from there:
    • subnet: 192.168.1.0/24
    • address: 192.168.1.333
    • gateway: 192.168.1.1
    • nameserver: 8.8.8.8
    • search domains: optional

    Warning

    can skip below now

    after installation - using netplan¶

    ip -c a

sudo mv 00-installer-config.yaml 00-installer-config.yaml.bak

sudo chmod 600 /etc/netplan/00-installer-config-wifi.yaml

sudo tee -a /etc/netplan/00-installer-config-wifi.yaml << EOF
network:
  version: 2
  renderer: networkd
  wifis:
    wlp2s0:
      dhcp4: false
      dhcp6: false
      addresses: [192.168.1.333/24]
      nameservers:
        addresses: [8.8.8.8] (will change later to 127.0.0.1 for pi-hole, so add rn)
      access-points:
        test:
          password: "bruh"
      routes:
        - to: default
          via: 192.168.1.1

EOF

sudo tee -a /etc/netplan/00-installer-config-wifi.yaml << EOF
network:
  version: 2
  renderer: networkd
  wifis:
    wlp2s0:
      dhcp4: on
      access-points:
        test:
          password: "bruh"
EOF


sudo netplan apply
ping google.com

    Note

    Netplan:

    • is a network configuration abstraction renderer used in modern Ubuntu releases
    • provides a high-level, YAML-based configuration interface for defining network interfaces and settings
    • generates network configuration files in the format understood by the underlying networking subsystem, which can be either:
    • systemd-networkd
    • NetworkManager

    Best practice:

    • on servers, always use systemd-networkd (baked into systemd)

    harden ssh¶

    sudo tee -a /etc/ssh/sshd_config <<EOF
Port 2059
AllowUsers dpi0
PubkeyAuthentication yes
PermitRootLogin no
PasswordAuthentication no
PermitEmptyPasswords no
EOF

sudo sed -i '/PasswordAuthentication/s/^/#/' /etc/ssh/sshd_config.d/*.conf
echo "PasswordAuthentication no" | sudo tee -a /etc/ssh/sshd_config.d/*.conf

sudo systemctl restart ssh

    setup ssh keys - on host¶

    ssh-keygen -t ed25519 -f ~/.ssh/home-server -C "home server"
ssh-copy-id -i ~/.ssh/home-server.pub dpi0@192.168.1.100

sudo chmod 600 ~/.ssh/home-server

    now passwords won't work and only way to ssh is to use a private key

    ssh -p 3333 -i ~/.ssh/home-server dpi0@192.168.1.333

    to not fill in passphrase everytime for the current shell session:

    • eval "$(ssh-agent -s)" --> ssh-add ~/.ssh/home-server
    • then enter passphrase
    • NOTE: this will persist only for the current shell session check by echo $SSH_AGENT_PID

    Note

    • RSA: ssh-keygen -b 4096 gives portability on older systems, Ed25519 is newer and stronger (no size needed as all of these are 256 bits)
    • If you need more security than RSA-2048 offers, the way to go would be to switch to elliptical curve cryptography Ed25519 and not 4096
    • ~/.ssh/known_hosts is your personal Certificate Authority. It is the list of all SSH server host public keys that you have determined are accurate.
    • to later add/change password on ssh-key: ssh-keygen -p -f ~/.ssh/your-key-filename

    more¶

    • a hardware clock, also known as RTC, CMOS or BIOS clock.
    • This is the battery-backed clock that keeps time even when the system is shut down.
    • The second clock is called the system clock/tick or 'kernel clock' and is maintained by the operating system.
    • At boot time, the hardware clock is read and used to set the system clock. From that point onward the system clock is used to track time.

    NTP¶

    • Network Time Protocol (NTP) is a set of rules/protocols, or conventions, that dictate how devices on a network should transmit, receive and understand data
    • NTP allows networked devices, such as clocks, phones and computers, to request and receive time from a server that, in turn, receives precise time from a definitive time source, like an atomic clock.
    • NTP makes UTC available to an organisation by taking a time signal from one, or more, atomic clocks and distributing it to networked devices.
    • an NTP server, which receives UTC time and provides it to the clients.
    • check status of time server: sudo systemctl status systemd-timesyncd.service
    • by def it uses ntp.ubuntu.com: Status: "Initial synchronization to time server 91.189.91.157:123 (ntp.ubuntu.com)."
